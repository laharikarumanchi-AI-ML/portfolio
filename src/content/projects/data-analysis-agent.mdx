## TL;DR

Data analysis questions — *"what's the median order value by city," "is there a relationship between tenure and churn"* — are everywhere in real work, and a poor fit for chat-only LLMs. This is a **code-as-action agent**: it answers questions about CSV data by writing and executing Python in a sandboxed Jupyter kernel, the same way an analyst would in a notebook. On a 9-task pilot of [InfiAgent-DABench](https://github.com/InfiAgent/InfiAgent) using Llama-3.3-70B via Groq, it scored **75% ABQ** under the official scorer. The pilot is honest about its scope — see [Evaluation](#evaluation).

> **75% ABQ on a 9-task pilot of InfiAgent-DABench (Llama-3.3-70B via Groq).** 9 of 80 attempted tasks completed before free-tier quota ran out; the remaining 71 hit `429 Too Many Requests`. The agent loop itself works (33 tests passing) — the bottleneck is API quota, not correctness.

## Problem

Tabular data questions are everywhere in real work. *"What's the median fare by region. Is there a relationship between tenure and churn. Show me the top five products by margin."* These are the questions a junior analyst gets asked in their first week on the job — and they're a surprisingly poor fit for chat-only LLMs.

The mismatch is structural. The model can't *see* the spreadsheet. You can paste a CSV header and a few sample rows into the chat, but that's a five-line teaser of a 50,000-row file. The model has to guess what's in the rest. It guesses how the categorical columns are spelled (is it `"Female"` or `"female"` or `"F"`?). It guesses whether there are nulls. It guesses what the column distribution looks like. Most of the time those guesses are reasonable — and that "most of the time" is exactly the problem. The wrong guess produces a confidently-presented wrong answer, and there's no way for the model to find out it was wrong, because it never actually touched the data.

The honest fix is to let the model touch the data. That means writing and executing code.

### Why ReAct tool-call agents struggle here

The standard playbook for "LLM that does things" is a ReAct loop with a predefined toolbox. LangChain, LlamaIndex, CrewAI, and most LangGraph examples wire it up the same way: the agent picks a tool (`pandas_read_csv`, `df_groupby`, `compute_mean`), the tool runs, the result comes back as an observation, and the agent picks the next tool. It's the right shape for *"book a flight or check the weather"* — a small, discrete action space.

It falls apart on multi-step pandas. A real analysis question — *"average tip percent by day-of-week"* — is read, derive a new column, groupby, format. Four pandas operations. In a tool-call agent, that's four full round-trips to the LLM, each with the previous observation re-fed as context, and each generating an entire tool-call schema response. Latency stacks. Token cost stacks. And the agent is reasoning at the wrong altitude — picking from a menu of `df_apply_function` shapes instead of writing the pandas chain the way an analyst would.

The deeper limitation is expressiveness. A tool-call menu can only let the agent do what the toolbox author thought to expose. Real pandas work is full of small, project-specific transformations (`df['tip_pct'] = df.tip / df.total_bill * 100`) that you'd never add as a permanent tool. A code-as-action agent doesn't have that ceiling — it writes the line directly.

### What I actually wanted

The model I had in my head was a junior analyst opening a notebook. They write a cell, look at the output, write the next cell. If the column is named `Fare` and they typed `fare`, they see the `AttributeError`, fix it, and move on. They don't ask permission first. They don't pick from a menu. They *think in code*.

That's a code-as-action loop: the model emits Python, the Python runs inside a live Jupyter kernel, the resulting stdout / stderr / display data / images get fed back as the next observation, and the loop repeats until the model emits a final answer. One model call per step, not one per pandas operation. The agent gets to use the full expressiveness of pandas — including ad-hoc lambdas, list comprehensions, and one-off helper functions — without having to ask anyone for permission first.

### Why InfiAgent-DABench specifically

I needed a benchmark that wouldn't let me lie to myself about how well the agent worked.

[InfiAgent-DABench](https://github.com/InfiAgent/InfiAgent) is a closed-form benchmark of 257 real data-analysis questions across a bundle of CSV tables. The questions look exactly like the kind of thing an analyst would actually be asked: *median fare on the Titanic dataset, correlation coefficient between two columns, top-N country by some aggregate metric.* Not "explain pandas in plain English." Real questions with real numerical answers.

Three properties made it the right fit. **First, the gold labels live in a separate file from the questions** — `da-dev-questions.jsonl` references tasks by ID; `da-dev-labels.jsonl` holds the answer schema. That's a small detail that turns out to matter: it forces the eval harness to do an explicit join, which makes it harder to accidentally peek at answers during agent execution. **Second, the official scorer extracts answers from a strict `@name[value]` tag schema** — `<answer>@mean_fare[34.65]</answer>` — using `re.findall(r"@(\w+)\[(.*?)\]", response)`, then matches the names against `common_answers` and runs a strict `is_equal` with float tolerance. **Third, the benchmark publishes a leaderboard** — so the numbers I report are directly comparable to other agents' numbers, instead of being a homegrown metric that nobody can sanity-check.

### What "honest evaluation" means here

The first temptation when evaluating any LLM was to write `predicted.lower() in expected.lower()` and call it 75% accuracy. That number is meaningless. The predicted string `"The mean is 34.65"` and the expected string `"34.65"` both pass; so does `"The mean is roughly 35"`, which is wrong. A substring check inflates scores by exactly the amount you don't want it to.

Instead, I copied InfiAgent's `eval_closed_form.py` into `agent/eval/scorers/infiagent/` verbatim, with attribution headers at the top of the file. The scorer extracts the `@name[value]` tags from the response, joins them against the gold `common_answers` by name, and runs strict equality with a small float tolerance. The trade-off is that the scorer is *stricter* than a naive matcher, so the numbers are lower than they would be otherwise — but they're honest, and they're leaderboard-comparable.

That's the whole reason this benchmark exists: to produce numbers that survive contact with someone else's evaluation infrastructure.

## Approach

The agent is built on a code-as-action loop, not a ReAct tool-call loop. Instead of exposing a fixed toolbox, the system prompt instructs the model to emit one of two things on every turn: a Python snippet wrapped in `<code>...</code>` to execute next, or a final answer wrapped in `<answer>...</answer>` when it has enough information. Everything else — the kernel, the message stream, the retry budget — exists in service of that single shape.

### The loop in plain English

1. The orchestrator reads `prompts/system.txt`, fills in the dataset path and a 5-row preview, and seeds the conversation with `[system, user]`.
2. The LLM responds. The orchestrator extracts whatever's between `<code>` and `</code>` (or `<answer>` and `</answer>`) with a small regex pair.
3. If the response is a code block, the orchestrator hands it to the sandbox.
4. The sandbox sends `execute_request` to a per-session Jupyter kernel via `jupyter_client`. It pumps the IOPub message stream until it sees `status: idle` or hits the per-cell deadline.
5. The orchestrator formats whatever came back — `stdout`, `stderr`, exception traceback, figure-count — into a single observation string and appends it as the next user-turn message.
6. The LLM responds again, with the next code block or the final answer.
7. The loop repeats up to `MAX_STEPS = 10`. The agent stops as soon as it emits `<answer>...</answer>`.

That's it. The whole orchestrator is about 150 lines of Python. No framework, no scheduler, no graph. The data dependencies are obvious because they're flat.

### Why no LangChain (or LlamaIndex, or CrewAI)

The first version of this project I sketched did import `langchain`. I deleted it within an afternoon. Not because the frameworks are bad — they're fine for the cases they're designed for — but because they hide exactly the behaviors I needed to debug.

The retry budget is one. The system prompt's exact format is another. The way malformed model output (a code block with no closing tag, an answer tag that's missing the `@name[value]` shape) gets handled is a third. With a 150-line orchestrator, all three are obvious from reading `orchestrator.py` start to finish. With a framework, the same behaviors are spread across config, callbacks, and subclass overrides, and you find out about them when something breaks at midnight.

This is a deliberate signal, not a value judgment of those frameworks. *"I want to understand my tools"* is one of the few things a portfolio project is actually useful for.

### Deliberate choices

A few non-obvious decisions shape what this agent can and can't do.

**A real Jupyter sandbox, not a subprocess.** It's tempting to write `subprocess.run(['python', '-c', code])` and call that a sandbox. It isn't — the subprocess has full host access, no incremental message stream, and no shared state between calls. `agent/sandbox.py` wraps `jupyter_client.KernelManager` instead. State persists across cells (just like a notebook), the IOPub message stream gives me incremental stdout / stderr / display data, and there's a single shared kernel per task that gets cleanly shut down at the end.

**A single deadline per cell, not a per-message timeout.** The cell timeout is 30 seconds by default. That budget is enforced at the loop level (`deadline = time.monotonic() + self._timeout`), not per `get_iopub_msg` call — otherwise a long-running cell that's actively streaming output would never time out, because every message would reset the timer.

**matplotlib figures captured via IOPub PNG.** The Jupyter kernel's default inline backend (`matplotlib_inline.backend_inline`) emits `image/png` data as `display_data` messages on `plt.show()`. The sandbox decodes those into raw PNG bytes and surfaces a figure-count to the model. (The system prompt explicitly tells the agent **not** to call `matplotlib.use('Agg')`, because forcing `Agg` suppresses the inline backend and the figures vanish silently.)

**A provider abstraction behind a one-method `LLMClient`.** Both Groq and Gemini implement the same `chat(messages, **kwargs) -> str` interface. The orchestrator never sees a provider name. Why this matters: when Groq's free-tier TPM ceiling started biting halfway through the first eval, swapping in Gemini was a five-line change in the eval runner, not a rewrite.

**The official InfiAgent scorer, copied verbatim with attribution.** `agent/eval/scorers/infiagent/eval_closed_form.py` is the upstream file with attribution headers at the top. Strict `is_equal` with float tolerance, no homegrown fuzzy matching. The numbers this produces are lower than a naive scorer's, but they're directly comparable to the DABench leaderboard.

### A 2-step trace from a real run

Task 0 in the eval run was *"Calculate the mean fare paid by the passengers"* over the Titanic table (`test_ave.csv`). The format constraint was `@mean_fare[value]`, rounded to 2 decimal places. The official scorer marked it correct; the trace shows 2 steps in 1.76 seconds wall-clock.

Reconstructed from the system prompt and the recorded final answer, the cells looked like this:

**Step 1 — the agent emits:**

```python
<code>
import pandas as pd
df = pd.read_csv(dataset_path)
print(round(df['Fare'].mean(), 2))
</code>
```

The sandbox executes the cell. The observation comes back as:

```text
[stdout]
34.65
```

**Step 2 — the agent emits the answer:**

```text
<answer>@mean_fare[34.65]</answer>
```

The orchestrator extracts `@mean_fare[34.65]`, records the final step, and returns. The scorer extracts `("mean_fare", "34.65")` from the answer string, joins against `common_answers = [("mean_fare", "34.65")]`, and calls `is_equal("34.65", "34.65")` — true.

Two model calls. One kernel execution. No tool-call schema. No framework. The model wrote the pandas chain directly because the agent loop got out of its way.
